# Git Worktrees — Hoops Dev Guide

> `git worktree` is a **git** feature, not a GitHub one. It lets one repository
> have multiple working directories checked out at once, each on its own branch,
> all sharing a single `.git` object store. No duplicated history, no extra clones.

The goal of this setup: spinning up a new worktree — including a dedicated one
per Claude Code session — should cost **one command**, with the only slow step
being `npm ci`.

---

## Why we use worktrees here

- **No stash-and-switch.** A prod bug surfaces mid-feature → `cd` into a worktree
  already on `main`/`hotfix`, fix it, come back. Your feature work never moves.
- **Parallel builds/tests.** Separate directories mean `dotnet build` and
  `ng build` don't fight over `bin/`, `obj/`, or `dist/`.
- **Local PR review.** Drop a teammate's branch in a throwaway worktree, run it,
  delete it — your own state is untouched.
- **Parallel agentic dev.** Run multiple Claude Code sessions at once, each in its
  own worktree on its own branch. Physically separate files = agents don't collide.

---

## The local-files problem (and why it's smaller than it looks)

The two halves of the stack behave very differently:

### .NET backend — ideally zero work

If dev secrets live in **`dotnet user-secrets`**, they sit _outside the repo_ at
`~/.microsoft/usersecrets/<UserSecretsId>/secrets.json`, keyed by the
`UserSecretsId` GUID in the `.csproj`. Since the `.csproj` is tracked, **every
worktree points at the same secret store automatically** — no linking, no copying.

> If dev config currently lives in a gitignored `appsettings.Development.json`,
> migrating it to user-secrets makes the worktree problem disappear for the
> backend entirely. Recommended.

### Angular frontend — a small, fixed list to link

This is where the real work is, but it's a _known, rarely-changing_ list of
gitignored files (e.g. `.env`, `proxy.conf.json`, `environment.local.ts`). Because
the list is fixed, it scripts cleanly — see the bootstrap script below.

**Symlink vs. copy:** symlink (`ln -s`) for stable config you want identical
everywhere (the default); copy (`cp`) only when a branch must diverge (e.g. points
at a different local DB). Caveat: tools that _atomically replace_ a file (delete +
rewrite rather than edit in place) will break a symlink — use copy for anything
regenerated by a build/codegen step.

## Layout

Keep the existing clone as the canonical "main" worktree; add siblings beside it.
The canonical clone is the **source of truth** for local/gitignored files.

```
~/dev/hoops/             <- existing clone, stays on main; source of local files
~/dev/hoops-feature-x/
~/dev/hoops-hotfix/
~/dev/hoops-claude-1/    <- dedicated to an agent session
```

## Bootstrap script: `wt-init.sh`

One command per new worktree. Adjust `LOCAL_FILES` and the dependency paths to
match the actual Hoops folder names.

```bash
#!/usr/bin/env bash
set -euo pipefail
# Usage: ./wt-init.sh <branch-name> [base-branch]

BRANCH="$1"
BASE="${2:-develop}"                       # Gitflow: branch off develop by default
REPO_ROOT="$(git rev-parse --show-toplevel)"
WT_PATH="${REPO_ROOT}/../hoops-${BRANCH}"
SRC="$REPO_ROOT"                           # canonical source for local files

# 1. Create the worktree on a NEW branch off base
git worktree add -b "$BRANCH" "$WT_PATH" "$BASE"

# 2. Link gitignored local files — ADJUST these paths to Hoops' actual layout
LOCAL_FILES=(
  "ClientApp/.env"
  "ClientApp/src/proxy.conf.json"
  "ClientApp/src/environments/environment.local.ts"
)
for f in "${LOCAL_FILES[@]}"; do
  if [ -e "$SRC/$f" ]; then
    mkdir -p "$WT_PATH/$(dirname "$f")"
    ln -sf "$SRC/$f" "$WT_PATH/$f"         # swap for cp if a file must diverge
    echo "linked $f"
  fi
done

# 3. Dependencies (backend restore is cheap; npm is the slow one)
( cd "$WT_PATH/Api"       && dotnet restore )
( cd "$WT_PATH/ClientApp" && npm ci )

echo "Ready: $WT_PATH"
```

You only edit `LOCAL_FILES` when a _new_ gitignored config file is added to the
project — rare. Day to day it's `./wt-init.sh feature-x` and you're working.

## Optional: fully hands-off via `post-checkout` hook

`git worktree add` fires the `post-checkout` hook, and hooks are shared across all
worktrees via the common git dir. A hook can auto-link files on creation:

```bash
#!/usr/bin/env bash
# .git/hooks/post-checkout
old_ref="$1"; branch_flag="$3"
ZERO="0000000000000000000000000000000000000000"
# Only fire on a fresh worktree/clone (old_ref all-zeros), not normal switches
[ "$branch_flag" = "1" ] && [ "$old_ref" = "$ZERO" ] || exit 0

main_wt="$(git worktree list --porcelain | awk '/^worktree /{print $2; exit}')"
here="$(pwd)"
[ "$here" = "$main_wt" ] && exit 0
for f in ClientApp/.env ClientApp/src/proxy.conf.json; do
  if [ -e "$main_wt/$f" ] && [ ! -e "$here/$f" ]; then
    mkdir -p "$(dirname "$here/$f")"
    ln -sf "$main_wt/$f" "$here/$f"
  fi
done
```

> Treat this as a phase-2 upgrade, not the starting point. `post-checkout` also
> fires on `git clone` and on every branch switch, so the all-zeros guard matters,
> and "first worktree in the list = canonical source" is a heuristic that holds for
> this simple layout but not universally. **Test it once** on a throwaway branch,
> confirm links appear, then `git checkout` around to confirm it does _not_ re-fire
> destructively. If it's fiddly, the explicit script is more predictable — and
> predictability beats saving one typed command.

## VS Code support

Native worktree support landed in **VS Code 1.103 (July 2025)** — no extension
required for the core workflow.

**Built-in actions:**

- Command Palette → `Git: Create Worktree`, `Git: Open Worktree in New Window`,
  `Git: Delete Worktree`.
- Source Control → **Repositories** view auto-detects linked worktrees and lets
  you create / open / switch / delete them. Each worktree shows as its own entry.

**The right mental model in VS Code:** open each worktree as its **own window**
(`Open Worktree in New Window`). Each window gets its own terminal, file tree,
extension host, and running dev server — which is exactly what makes a per-window
Claude Code session clean. A multi-root workspace can hold several worktrees in one
window, but separate windows keep the agent sessions and dev servers isolated.

**Still terminal-only:** advanced flags like `--lock`, `--reason`, and
`git worktree repair`. Day-to-day create/open/delete is all in the UI.

**Extensions (optional):** GitLens adds a richer worktrees panel (blame, commit
graph, comparisons) if you already use it; otherwise the built-in view is enough.

## Merging to `develop` from a worktree

Short version: **easy, with one rule to internalize** — _a branch can only be
checked out in one worktree at a time._

Because all worktrees share the same object store, a feature branch's commits are
**instantly visible from every worktree** — no `fetch`, no `pull`, no syncing
between them. The merge just needs to run in the worktree that holds the _target_
branch.

### The clean pattern (Gitflow: `feature-x` → `develop`)

Your canonical `~/dev/hoops` clone stays on `develop`. So:

```bash
cd ~/dev/hoops          # this worktree is already on develop
git merge feature-x     # feature-x's commits are already present locally
```

That's it. You never had to leave the feature alone or stash anything in the
feature worktree — it stays exactly as it was.

### The error you'll hit once, and why

From _inside_ the feature worktree, this fails:

```bash
git switch develop
# fatal: 'develop' is already checked out at '/home/you/dev/hoops'
```

That's git protecting you: `develop` is live in another worktree, so it won't let
a second one grab the same branch. The fix is conceptual, not mechanical — **don't
bring `develop` to the feature; take the feature to `develop`.** Merge in the
worktree that already has `develop` (above).

### In practice you'll usually PR it

For Gitflow with CI/CD, the normal path is: push the feature branch, open a PR into
`develop`, let GitHub Actions run. Worktrees don't change that at all — they just
make the _local_ verify-and-merge step painless when you do it by hand. VS Code's
Source Control view can stage/commit/merge in whichever worktree window holds
`develop`, following the same one-branch-one-worktree rule.

## Daily commands

```bash
./wt-init.sh hotfix-scoring-bug main        # spin up (hotfix off main)
./wt-init.sh feature-x                       # spin up (feature off develop)
git worktree list                            # see everything
git worktree remove ../hoops-feature-x       # clean up when merged
git worktree prune                           # sweep stale entries after a manual rm -rf
```

### Two gotchas (each bites once, then never again)

1. A branch can only be checked out in **one** worktree at a time. That's why
   `wt-init.sh` creates a _new_ branch with `-b` off the base rather than checking
   out the base itself.
2. If you `rm -rf` a worktree dir instead of `git worktree remove`, git keeps a
   dangling admin entry until you `git worktree prune`.
